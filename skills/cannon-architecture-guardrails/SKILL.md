# Cannon architecture guardrails

Use this skill whenever work could produce or modify a modern raid `.schem` or `.litematic`.

## First principle

A real modern cannon is a three-dimensional impulse machine. Redstone schedules entity creation. TNT cohorts, payloads, chambers, fluids, hammers, splitters, stoppers, guiders, pistons, alignment, gravity, collision and order-of-entities perform the physical work.

Never replace that machine with flat dispenser rows or a visually tidy redstone platform.

## Mandatory rules

1. Begin from an exact decoded reference or bounded variant. Blank-canvas work is a `diagnostic-prototype`, never a modern raid candidate.
2. Run the geometry-profile morphology gate. A failed anti-pancake verdict blocks the candidate.
3. Declare the stage graph and physical impulse edges. Identify which stage pushes, redirects, aligns or transfers the next stage.
4. Treat static role names as candidates only. Confirmed roles require runtime or field evidence.
5. Declare one causal variable and one touched module by default. Broader changes require a written override.
6. Preserve exact reference geometry outside the declared change budget and require a passing preservation report.
7. Promotion must use native redstone. Direct dispense, forced velocity, TNT probes, simulated durability and suppressed paste side effects are diagnostic-only.
8. Require passing payload, trajectory, breach, survival and self-damage acceptance evidence before `working`, `fixed` or `one-shot` language.
9. Require an explicit one-shot contract for a `one-shot` claim.
10. Require a passing live ExtremeCraft canary plus `field_verified=true` before `EC-ready`.

## Required command

Before exporting or promoting a candidate, validate its architecture manifest:

```powershell
python scripts/validate-cannon-architecture.py path/to/architecture-manifest.json
```

Read `policy/README.md` for the manifest shape and `policy/modern-cannon-architecture-policy.json` for the machine-checkable rules.

A failed validator result is a blocker, not a suggestion.

## Truth boundary

A policy pass proves that the evidence package is complete enough for its declared lifecycle. It does not independently prove the cannon works or matches private ExtremeCraft mechanics. The runtime and field evidence still carry those claims.
